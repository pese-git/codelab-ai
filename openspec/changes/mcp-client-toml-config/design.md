## Context

Клиент ACP в настоящее время не загружает конфигурацию MCP серверов из TOML файлов. При создании или загрузке сессии клиент передает пустой список `mcpServers=[]` серверу, хотя:
- ACP протокол поддерживает параметр `mcpServers` в `session/new` и `session/load`
- Серверная часть полностью реализует MCP интеграцию через `MCPManager`
- Пользователи уже используют TOML файлы для конфигурации LLM провайдеров и TUI

Текущая архитектура клиента следует Clean Architecture с 5 слоями: Domain → Application → Infrastructure → Presentation → TUI.

## Goals / Non-Goals

**Goals:**
- Клиент загружает MCP серверы из TOML файлов при старте
- MCP серверы передаются серверу при создании/загрузке сессии
- Поддержка env var expansion во всех строковых полях
- Graceful degradation при невалидной конфигурации
- Независимость от серверных модулей

**Non-Goals:**
- Не добавляет поддержку новых типов транспортов MCP (sse уже поддерживается сервером)
- Не реализует MCP клиент на стороне клиента (это остается на сервере)
- Не изменяет ACP протокол
- Не добавляет UI для управления MCP серверами

## Decisions

### Decision 1: MCPConfigLoader в Infrastructure слое

**Решение:** Разместить `MCPConfigLoader` в `client/infrastructure/mcp_config_loader.py`

**Почему:** 
- Загрузка конфигурации — инфраструктурная задача
- Аналогично `TUIConfigStore` в `client/tui/config.py`
- Не зависит от Domain/Application логики

**Альтернативы:**
- Разместить в `client/tui/` — слишком специфично для TUI, не подходит для других клиентов
- Разместить в `client/application/` — смешивает инфраструктуру с бизнес-логикой

### Decision 2: Override по name при merge TOML

**Решение:** При объединении `[[mcp.servers]]` из нескольких TOML файлов, серверы с одинаковым `name` переопределяются (файл с высшим приоритетом побеждает)

**Почему:**
- Предсказуемое поведение — пользователь знает что `codelab.local.toml` переопределит `codelab.toml`
- Соответствует паттерну TOML array of tables
- Проще чем append (нет дубликатов)

**Альтернативы:**
- Append всех серверов — возможны дубликаты, сложнее управлять
- Merge по всем полям — слишком сложно, неочевидно

### Decision 3: Env var expansion во всех строковых полях

**Решение:** Рекурсивная функция `expand_env_vars()` применяется ко всем строковым полям: `command`, `args[]`, `url`, `headers[].value`, `env[].value`

**Почему:**
- Консистентность — пользователь ожидает что `${VAR}` работает везде
- Безопасность — API keys и токены обычно в env vars
- Аналогично серверному `toml_loader.py:expand_env_vars()`

**Альтернативы:**
- Только в sensitive полях (headers, env) — менее гибко
- Только в command — недостаточно для MCP серверов с динамическими URL

### Decision 4: Silent skip невалидных серверов

**Решение:** Невалидные MCP серверы пропускаются с warning в лог, но не прерывают загрузку

**Почему:**
- Graceful degradation — один невалидный сервер не ломает остальные
- Соответствует поведению серверного `MCPManager.add_server()` (try/except с continue)
- Удобно для development — можно иметь частичную конфигурацию

**Альтернативы:**
- Error при загрузке — слишком строго, ломает весь MCP
- Error только для missing name — непоследовательно

### Decision 5: Независимость от серверных модулей

**Решение:** `MCPConfigLoader` не импортирует модули из `codelab.server.*`

**Почему:**
- Clean Architecture — клиент не зависит от сервера
- Уменьшает coupling
- Позволяет использовать клиент отдельно

**Альтернативы:**
- Использовать серверный `TOMLConfig` — создает зависимость client → server
- Дублировать код — приемлемо для такой небольшой функциональности

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| TOML файлы содержат несовместимые MCP серверы | Валидация на клиенте, silent skip с logging |
| Env vars не установлены в runtime | Expansion в пустую строку, сервер валидирует конфигурацию |
| Пользователь ожидает что MCP серверы автоматически подключатся | Документация, логирование количества загруженных серверов |
| Конфликт имен серверов между TOML файлами | Override по name — документированное поведение |
| Производительность при большом количестве TOML файлов | TOML chain обычно 2-4 файла, загрузка при старте (не критично) |

## Migration Plan

1. Добавить `MCPConfigLoader` — без влияния на существующий функционал
2. Расширить `ClientConfig` — новое поле с default `[]` (backward compatible)
3. Обновить UseCase и Coordinator — новые параметры optional
4. Обновить TUI App — загрузка MCP конфига при старте
5. Обновить `codelab.toml.example` — добавить примеры

**Rollback:** Удалить изменения, клиенты продолжат передавать `mcpServers=[]`

## Open Questions

1. **Нужна ли поддержка SSE транспорта в клиентской конфигурации?**
   - Сервер поддерживает, но SSE deprecated в MCP spec
   - Решение: поддержать через `type = "sse"` для обратной совместимости

2. **Нужен ли UI для отображения загруженных MCP серверов?**
   - Не в scope этого изменения
   - Можно добавить позже через tool panel

3. **Нужна ли валидация JSON Schema для args/env?**
   - Нет — сервер валидирует при инициализации MCP серверов
